# 🎯 pi-mojo v0.2: Comprehensive Systems Development Plan (Updated Layout)

This document presents the finalized, in-depth architectural and development blueprint for the **`pi-mojo` v0.2** release. Based on design feedback, this iteration separates core capabilities from real-world operations workflows:

1. **Modular Higher-Level Capabilities**: 
   * **Autonomous Playbooks (`src/packages/playbook`)**: A dual-loop system allowing agents to compile successful runs into reusable Markdown/YAML recipes, optimizing future runs.
   * **Durable Session Checkpointing (`src/packages/durable`)**: A crash-resilient checkpointing store utilizing Python `json` interop bindings for bulletproof state serialization.
   * *Note: No Model Context Protocol (MCP) tool integration will be introduced, preserving `pi-mojo`'s compact and lightweight systems-focused footprint.*
2. **Dedicated `scenarios/` Directory**:
   * Instead of listing scenario-driven examples inside `examples/` or the root `README.md` directly, a brand new top-level **`scenarios/`** directory will host all 10 scenario-driven systems examples.
   * The root `README.md` will contain a brief section towards the end pointing to the `scenarios/` directory.
   * The `scenarios/` directory will feature its own dedicated `README.md` describing the design philosophy of the storyboards.
   * Each scenario will occupy its own sub-folder containing its Mojo code, a scenario-specific `README.md`, and recorded run outputs.

## 🛠️ Pillar 1: Modular Higher-Level Capabilities

### 📘 A. Autonomous Playbook Learning (`src/packages/playbook`)
This package equips agents with a **double-loop memory system** (similar to advanced software engineering agents). 

#### 1. Playbook Schema (`playbook.md`)
Playbooks are compiled into clean Markdown files inside a `.pi_playbooks/` folder. They feature a YAML frontmatter header followed by a step-by-step procedural recipe:
```markdown
# Playbook: [Goal Description]
- **Tags**: [comma-separated-tags]
- **Criteria**: [definition-of-done]

## Steps
1. `[command_or_action_1]`
2. `[command_or_action_2]`
```

#### 2. Architecture Specifications
* **`pi_playbook_store.mojo`**: 
  * Manages `.pi_playbooks/` namespace operations.
  * Implements `load_playbooks() -> List[Playbook]` using `readdirSync` from `t2m_runtime.fs` and standard Python `open` interop.
  * Parses frontmatter variables and steps from Markdown using robust Python-based regex operations to avoid native string allocation overhead.
  * Implements `find_matching_playbook(task: String) -> Playbook` which runs SIMD-like string match scanning across playbook tags and goal descriptions.
* **`pi_playbook_agent.mojo`**:
  * Inherits and wraps the core `PiAgent` execution loops.
  * **Alignment Turn**: Prior to initiating a new task, it queries the store for a matching playbook. If a recipe is found, it prepends the successful steps into the system context: `"You have successfully completed a similar task before. Follow this recipe: ..."`
  * **Synthesis Turn**: If the task completes with a successful status, it triggers a lightweight post-run reflection request to the LLM. The LLM extracts the exact execution path into the markdown playbook format, and the store persists it.

### 💾 B. Durable Session Checkpointing (`src/packages/durable`)
To support crash resilience, manual loop pauses, or multi-day runs, this package persists the agent's exact execution state machine to disk under `.pi_checkpoints/`.

#### 1. Checkpoint Schema (`.json`)
State is saved as a structured JSON snapshot:
```json
{
  "session_id": "session_durable_demo",
  "goal": "Migrate INI files to JSON structure",
  "current_step": 2,
  "messages": [
    {"role": "user", "content": "Start config migration"},
    {"role": "assistant", "content": "Checking current directory structures."}
  ],
  "metadata": {
    "model": "google/gemini-3.5-flash",
    "timestamp": 1716634288000
  }
}
```

#### 2. Architecture Specifications
* **`pi_checkpoint_store.mojo`**:
  * Manages the `.pi_checkpoints/` repository.
  * To guarantee maximum compatibility and avoid Mojo interop string truncation, all JSON reading, parsing, writing, and directory checks are executed via Python interop (`json`, `os`, and `builtins` Python modules).
  * **`save_checkpoint(checkpoint: Checkpoint) raises`**: Maps the Mojo struct fields into a Python dictionary, serializes it using `json.dumps(d, indent=2)`, and commits it to disk using `builtins.open`.
  * **`load_checkpoint(session_id: String) raises -> Checkpoint`**: Reads the file contents using Python interop, deserializes via `json.loads`, extracts fields, and returns a rehydrated `Checkpoint` struct.
  * **`exists_checkpoint(session_id: String) raises -> Bool`**: Checks if `<session_id>.json` exists.
* **`pi_durable_agent.mojo`**:
  * Wraps agent execution structures.
  * Exposes `run_durable_task(session_id: String, goal: String, max_steps: Int) raises`.
  * Checks for checkpoint existence. If found, it instantly rehydrates history, step count, and system state, resuming exactly where it crashed.
  * Commits the current state to disk after every single tool call or model completion turn.

## 📖 Pillar 2: Dedicated `scenarios/` Systems Storyboards

The 10 new scenarios provide highly realistic systems-engineering contexts. They complement the corresponding capability examples by elevating them into narrative-driven, end-to-end automations.

---

### 🤖 Scenario 1: The AI Systems Onboarding Assistant
* **Folder**: `scenarios/scenario_1_onboarding_assistant`
* **Complements**: `examples/example_1_basic_ai` (Progressive AI Completions)
* **Narrative**: A newly hired engineer needs to onboard onto a development server. The agent must verify if required environments are configured, check shell profiles, and output a dynamic system onboarding diagnostics summary.
* **Design & Logic Flow**:
  1. Spawns an agent loop that runs an environment inspection.
  2. Queries the OS environment using Python interop (`os.environ`) for typical paths (`PATH`, `HOME`, `SHELL`, `LANG`).
  3. Formats an onboarding checklist, identifying missing variables, and generates a recommended `.bashrc` or `.zshrc` block to add.
  4. Saves a diagnostics summary report: `onboarding_diagnostics.txt`.
* **Output**: A comprehensive terminal log containing the onboarding report and system health checklist.

---

### 💻 Scenario 2: The Automated Git Repository Sanitizer
* **Folder**: `scenarios/scenario_2_git_sanitizer`
* **Complements**: `examples/example_2_coding_agent` (Systems Coding Agent)
* **Narrative**: A repository contains scattered temporary build artifacts, log dumps, and unstaged lockfiles. The agent autonomously inspects the workspace, cleans up untracked temporary files, and stages valid files.
* **Design & Logic Flow**:
  1. The agent executes a workspace scan using Python interop `subprocess` calls.
  2. It runs `git status --porcelain` to collect unstaged changes.
  3. It identifies temporary lockfiles (e.g., `*.lock`, `*.log`, `*~`) that should not be staged.
  4. Natively deletes temporary files, updates `.gitignore` if necessary, and clean-stages valid files (`git add`).
* **Output**: A clean git repository status and a deletion report of temporary artifacts.

---

### 🔧 Scenario 3: The Operations Thermal Stress Monitor
* **Folder**: `scenarios/scenario_3_thermal_stress`
* **Complements**: `examples/example_3_tool_calling` (Native Tool Calling)
* **Narrative**: An agent acts as a datacenter operations monitor. It calls native Mojo mathematical tools to compute stress factors based on incoming simulated sensor reports, outputting cooling load recommendations.
* **Design & Logic Flow**:
  1. Exposes two native Mojo functions as tools:
     * `get_datacenter_sensor_metrics() -> String` (returns simulated thermal, humidity, and fan speeds).
     * `calculate_cooling_load(temp: Float, power: Float) -> Float` (performs thermodynamic calculations natively).
  2. The agent queries these tools to evaluate simulated stress levels.
  3. If stress thresholds are exceeded, the agent outputs recommended actions (e.g., "Shed CPU load", "Throttle server fan speeds").
* **Output**: Step-by-step operations logs demonstrating tool execution and computed cooling rates.

### 🌊 Scenario 4: The Continuous Log Auditor Stream
* **Folder**: `scenarios/scenario_4_log_stream_auditor`
* **Complements**: `examples/example_4_event_stream` (Real-Time AI Streaming)
* **Narrative**: The agent monitors high-volume system logs (syslogs) in real time. It streams tokens from Gemini, highlighting severe security categories (e.g., `AUTH_FAIL`, `EXPLOIT_ATTEMPT`) dynamically as they appear.
* **Design & Logic Flow**:
  1. Generates a mock high-frequency stream of syslog entries (containing random standard logins, database queries, and simulated ssh brute-force entries).
  2. Feeds this active event stream through a streaming AI pipeline.
  3. Uses Mojo interop to output immediate color-coded console indicators matching specific severity indexes.
* **Output**: A live-updating terminal dashboard showing raw logs next to real-time AI security audits.

### 🔍 Scenario 7: The Lifecycle Leak & File-Descriptor Auditor
* **Folder**: `scenarios/scenario_7_lifecycle_auditor`
* **Complements**: `examples/example_7_codebase_auditor` (Semantic Codebase Auditor)
* **Narrative**: A project is experiencing resource leaks from unclosed descriptors. The agent crawls the codebase, isolates zero-copy `StringView` slices containing unmanaged resource allocation lifecycles, and creates a patching plan.
* **Design & Logic Flow**:
  1. Sets up a mock codebase containing unclosed handles (e.g., files opened but not closed inside exception branches).
  2. Runs a fast regex-based semantic scan across code paths.
  3. Identifies lines containing unmanaged resources.
  4. Generates a comprehensive refactoring patch plan to fix code lifecycles.
* **Output**: A detailed report pinpointing the exact file line ranges of resource leaks, accompanied by proposed code diffs.

### 🔁 Scenario 8: The Legacy Code Migration Assistant
* **Folder**: `scenarios/scenario_8_migration_assistant`
* **Complements**: `examples/example_8_long_running_coder` (Long-Running Coder Agent)
* **Narrative**: An engineering team is updating deprecated signatures in their codebase. The agent runs in a compilation loop, capturing compiler diagnostics and correcting deprecated interfaces step-by-step until the build compiles.
* **Design & Logic Flow**:
  1. Creates a target directory containing files with deprecated method calls.
  2. Spawns a background build loop.
  3. Parses compilation error logs returned by Mojo / Python build runners.
  4. Operates iteratively: patching syntax, executing builds, analyzing logs, and repeating until a successful compilation status is achieved.
* **Output**: A successful, clean compilation run accompanied by step-by-step syntax patch notes.

### 💓 Scenario 9: The Cluster Database Recovery Monitor
* **Folder**: `scenarios/scenario_9_db_recovery_monitor`
* **Complements**: `examples/example_9_local_heartbeat` (Local Service Heartbeat)
* **Narrative**: A critical database port is unresponsive. The agent conducts active latency probes and query timing tests. If database query timeouts are exceeded, it executes a multi-step database recovery playbook natively.
* **Design & Logic Flow**:
  1. Probes a local service.
  2. Simulates connection latencies and unresponsiveness.
  3. Triggers database diagnostic checks.
  4. If status is critical, it executes a recovery script (simulating db service restarts, log rotation, and connection pool flushing).

### 🔄 Scenario 10: The Continuous Integration Self-Healing Daemon
* **Folder**: `scenarios/scenario_10_ci_self_healer`
* **Complements**: `examples/example_10_full_fledged_agent` (Autonomous Loop Daemon)
* **Narrative**: A daemon agent runs in the background monitoring a workspace build log. If an integration build fails, the daemon intercept-analyzes the compiler errors, patches the target source files, and verifies the CI pass status.
* **Design & Logic Flow**:
  1. Sets up a ticking background timer (simulated daemon thread).
  2. Monitors a target file (build output).
  3. When a build failure is detected, the daemon wakes up, analyzes the code failure context, edits the offending code lines, and verifies the CI status passes.
* **Output**: Background daemon activity trail logs detailing build failure intercepts, auto-healing patches, and final clean CI status.
