Standard MoE routing pipelines suffer from severe performance degradation due to decoupled execution stages: a gating GEMM, followed by high-latency global/shared memory roundtrips for Softmax, Top-$K$ selection, index calculation, and a final token gathering/scattering stage.

## 2. Evolution of Technical Optimizations

### Phase 1: TMA Staging, 2D Warp-Cooperative Gemm, and Register-Level Fused Epilogue
* **Latency Achieved:** $208.4\ \mu\text{s}$

To overcome register starvation and latency on Blackwell GPUs, we targeted the decoupled load-store pipeline. Our Phase 1 architecture leverages the physical **Tensor Memory Accelerator (TMA)** to asynchronously pull token features ($X$) and gating projection weights ($W$) directly from global memory into Shared Memory (SMEM). This completely bypasses the register file during the initial data loading phase, minimizing register pressure and allowing maximum thread-block occupancy (up to 32 active warps per SM).

```
[Global Memory] ──(TMA Async Push / Zero RF Occupancy)──► [Shared Memory Scratchpad (128B Aligned)]
```

#### Fused 2D Warp-Cooperative GEMM & Epilogue
We deployed a **2D-tiled warp-cooperative layout** that computes the gating matrix multiplication ($G = X \cdot W$) using FP8 Tensor Cores. The resulting output matrix is consumed immediately in-flight by a custom fused epilogue. 

Rather than spilling the gate outputs back to global or shared memory, the epilogue performs Softmax and Top-$K$ selection directly inside the register file using high-speed **warp-shuffle intrinsics (`__shfl_xor_sync`)**. The tokens are quantized to FP8 on-the-fly, utilizing the register data directly.

To prevent SMEM bank conflicts during token routing and expert index assignment, we designed the SMEM scratchpad layout with a strict **128-byte padded alignment**, directly matching Blackwell’s L2 sector and cache line boundary. 

Finally, the routed FP8 outputs are committed to global memory using coalesced **128-bit vectorized write operations (`STG.128`)** with fused hardware-accelerated scaling factors.

### Phase 2: Warp-Aggregated Index Scans, XOR-Swizzling, and In-Register Packed FP8 Comparisons
* **Latency Achieved:** $138.2\ \mu\text{s}$

Phase 2 resolved two primary bottlenecks: atomic serialization during index allocation (where multiple threads attempt to write to the same expert slot) and bank conflicts during scatter/gather routing operations.

```
       Warp-Cooperative Scan Pipeline (Zero Memory Serialization)
       
       Thread ID:  [ T0 ]    [ T1 ]    [ T2 ]    [ T3 ] ... [ T31 ]
       Expert ID:  [  5 ]    [  5 ]    [  2 ]    [  5 ] ... [  2  ]
                   └──────┬──────┘              │           │
       __ballot_sync      ▼                     ▼           ▼
       Bitmasks:   [ 110100...0 ]        [ 001000...1 ] ... [ 001000...1 ]
       __popc      ▼                     ▼           ▼
       Reg Offset: [  0 ]    [  1 ]              [  0 ]      [  1 ]
                   │
                   ▼ (Leader Thread T0 Only)
       atomicAdd(&expert_count[5], 3) ──► Reserved Global Slot
```

#### Warp-Aggregated Cooperative Scans
To eliminate high-latency atomic contention in SMEM or global memory, we implemented a register-only **warp-level cooperative scan**. Individual threads do not issue independent atomic operations. Instead:
1. Threads inside a warp build a bitmask of peer threads targeting the same expert using `__ballot_sync`.
2. Threads calculate their localized routing offsets within the warp using the population count instruction (`__popc`) on the lane-masked result.
3. A single designated **leader thread** per warp executes a single collective `atomicAdd` to reserve slot space for the entire warp. 
4. The leader thread broadcasts the base offset back to the warp using `__shfl_sync`, allowing all lanes to compute their exact target addresses in parallel without any serialized conflicts.

#### 2D XOR-Swizzled Shared Memory Layout
We eradicated bank conflicts during the gather/scatter phase of dynamic routing by implementing a **2D XOR-swizzled shared memory layout**. The layout maps directly to the `cutlass::layout::RowMajorTensorOpMultiplicandCongruous` scheme, modified with a hardware-aligned 128-bit swizzle pattern. This guarantees that even under highly skewed token-to-expert assignments, threads within the same warp access unique physical memory banks:

$$\text{SharedAddress}(i, j) = \text{Base} + (i \times \text{Pitch}) + \left(j \oplus \text{SwizzleMask}(i)\right)$$

#### Direct Packed FP8 PTX Comparators
Instead of unpacking FP8 values to standard FP16 or FP32 types for Top-$K$ comparison operations (which bloats register usage and adds translation latency), we wrote inline PTX to use byte-permutation (`prmt`) and sub-word comparison instructions. These operators execute comparison logic directly on packed 4xFP8 registers, reducing instructions and saving precious register file space.

```cuda
// Inline PTX showing direct byte-level permutation and comparison packing
asm volatile(
    "prmt.b32 %0, %1, %2, 0x0040;\n\t" // Pack byte components directly
    : "=r"(packed_result) 
    : "r"(val_a), "r"(val_b)
);
```

### Phase 3: Template-Metaprogrammed Static Layouts, Alignment Guardrails, and Async Pipeling
* **Latency Achieved:** $82.5\ \mu\text{s}$

Phase 3 pushed our kernel to the physical limits of Blackwell's execution engines. We introduced a **Template-Metaprogrammed Static Layout and Alignment Strategy** that shifts all runtime index arithmetic, instruction overhead, and dynamic scheduling decisions to the compiler.

```
+-------------------------------------------------------------------------------------------------+
|                               Compiler-Time Specialization Layer                                |
+-------------------------------------------------------------------------------------------------+
|   Top-K (K ∈ {1, 2, 4, 8})               ──►   Completely Unrolled Sorting Networks             |
|   Experts (E ∈ {16, 32, 64, 128, 256})   ──►   Modulo & Div converted to Bitwise AND/Shifts     |
|   Expert Allocation Mask (E ≤ 32)        ──►   Single 32-bit Register Execution                 |
+-------------------------------------------------------------------------------------------------+
```

#### 1. Compile-Time Parameters & Dynamic Kernel Dispatch
We eliminated dynamic loops, branching, and register spilling by specializing the kernel along discrete compile-time bounds:
* **Top-$K$ ($K$):** Specialized for $K \in \{1, 2, 4, 8\}$. This allows the compiler to fully unroll sorting networks, converting dynamic register-indexed arrays into static, scalar register variables.
* **Expert Count ($E$):** Specialized for power-of-two bounds $E \in \{16, 32, 64, 128, 256\}$. This converts expensive runtime modulo (`%`) and integer division (`/`) instructions into cheap bitwise operations (`&` and `>>`).
* **Warp-Aggregation Bitmask Specialization:** When $E \le 32$, the expert allocation mask fits entirely into a single 32-bit register (one bit per expert). This allows warp-aggregation via `__ballot_sync` to run in a single clock cycle, completely bypassing SMEM-based tracking tables.

#### 2. Static 128-Bit Layout Alignments & Vectorized I/O Guardrails
To maximize memory bus utilization, we enforce a strict 16-byte (128-bit) physical boundary alignment across all input and output tensors via template assertions (`static_assert`).
* **Vectorized Data Transport (`LDG.128` / `STG.128`):** Ensuring that the hidden dimension $D$ is a multiple of 16 elements (for FP8) forces the compiler to emit vectorized 128-bit load/store instructions (`uint4` or `copy_128b`).
* **Aligned Pointer Wrappers:** All raw pointers are wrapped in alignment-annotated structures, ensuring the compiler bypasses alignment checks and schedules dual-issue instructions aggressively:

```cuda
template <typename T, int Alignment = 16>
struct AlignedPtr {
    using ElementType = T;
    static constexpr int kAlignment = Alignment;
    T* __restrict__ ptr;
    
    __device__ __forceinline__ uint4 load_vectorized(int offset) const {
        return *reinterpret_cast<const uint4*>(ptr + offset);
    }
};
```

#### 3. Meta-Swizzled Shared Memory Layout Instantiation
We instantiate a meta-swizzled SMEM layout using compile-time token dimensions and block sizes. The XOR-swizzle shift values and XOR-masks are computed programmatically at compile time based on the pitch of the SMEM tile. As a result, the address calculation compiles down to a single SASS-level ALU sequence of `shl`, `xor`, and `add` instructions, completely avoiding dynamic arithmetic during token routing.

#### 4. Async Copy (`cp.async` / `cp.async.bulk`) Multi-Stage Pipeling
By combining compile-time bounds and aligned layouts, we developed a multi-stage software-pipelined data-loading path. We configure the pipeline depth (e.g., 2-stage double buffering or 3/4-stage multi-buffering) as a `constexpr` parameter.

Combined with vectorized alignments, `cp.async.bulk` transfers tokens from global memory directly into the XOR-swizzled SMEM scratchpad. This allows the Streaming Multiprocessor (SM) to focus entirely on warp-aggregated routing calculations and Top-$K$ comparison networks without wasting instruction cycles on data movement.
